The idea

One spec in. Four environments out. A probe judges them.

There is no single “correct” environment for a spec, so we don’t grade against an answer key. Instead, building an environment is itself the task — modeled as a HUD environment whose agent is a builder model and whose reward comes from a nested HUD eval.

The reward of the outer environment is produced by running a full HUD evaluation inside its grader. That nesting is the “ception” — and it runs, live.

The pipeline

How a builder earns its score

1

Spec → builders

One rough JSON spec is handed to every builder model: GPT (gpt-5.5), Qwen (qwen3-coder), Sonnet (claude-sonnet-4-6), and the golden author Claude Opus 4.8.

2

Each builds a HUD environment

The builder returns a runnable module — an Environment, @env.template tasks, and a tasks=[…] list. The grader validates it loads.

3

A probe is evaluated on it

Inside the grader, a probe agent runs the freshly built environment 5× and the rewards are averaged — a nested HUD eval producing the outer reward.

4

Phase 2 · train, then judge on the golden env

A trainee will be RL-trained on each builder’s env, then scored on Opus’s golden benchmark. Whichever env trains the best trainee wins. blocked · Tinker

The artifacts

Exactly what each builder wrote

Same spec, four different environments. This is the real, unedited code each model submitted in the run above — and it explains the scores.

claude-opus-4-8 golden author
0.673 tasks
from hud import Environment


env = Environment(name="letter-count")


@env.template()
async def letter_count(word: str = "strawberry", letter: str = "r"):
    answer = yield f"How many '{letter}'s are in '{word}'? Reply with just the number."
    try:
        value = int(str(answer).strip())
    except (ValueError, TypeError):
        yield 0.0
        return
    yield 1.0 if value == word.lower().count(letter.lower()) else 0.0


tasks = [
    letter_count(word="strawberry", letter="r"),
    letter_count(word="mississippi", letter="s"),
    letter_count(word="banana", letter="a"),
]

Why it led: generalized one example into three varied test cases, parsed safely, and graded case-insensitively — the most thorough, hardest-to-fluke environment.

from hud import Environment


env = Environment(name="letter-count")


@env.template()
async def count_letter(word: str = "strawberry", letter: str = "r", expected: int = 3):
    prompt = f"How many '{letter}'s are in '{word}'? Reply with just the number."
    answer = yield prompt
    try:
        count = int(answer.strip())
        yield 1.0 if count == expected else 0.0
    except (ValueError, AttributeError):
        yield 0.0


tasks = [
    count_letter(word="strawberry", letter="r", expected=3),
]

Solid: wraps parsing in try/except so a malformed answer scores 0 instead of crashing — but only one test case.

Reality check

What the live infrastructure actually allows

The design is honest about HUD’s current limits — substitutions are one constant in config.py, swappable the moment the backend recovers.

✕ Gemma 4 can’t be trained

is_trainable: false on HUD, and gemini agent-type. The intended trainee can’t actually be RL-trained — so Phase 1 uses a working probe instead.

✕ Tinker backend is down

Every trainable model (Qwen3-8B, Llama-3.2-3B, gpt-oss…) routes through Tinker, which 400s with “SDK version no longer supported.” Blocks training and sampling those models.

✓ Nested eval works

Running a full HUD evaluation inside a HUD env’s grader — the core risk — was verified end-to-end on the gateway. The “ception” is real, not a slide.

✓ Builder = agent works

Each builder is an ordinary HUD agent, so the whole thing runs as a standard hud eval with traces on hud.ai — and a builder could later be RL-trained too.
